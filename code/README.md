# Project Code Description

This repository provides the released code components for the study entitled `Risk-Aware Intelligent Dispatching for Ride-Hailing Systems Using Dual-Graph Attention Forecasting and Unbalanced Optimal Transport`. The code is organized around two main modules: the CC-STMT prediction module and the DP-OTM dispatching module.

The released files are intended to document the implementation of the proposed prediction-dispatching framework and to support reproducibility at the methodological and experimental-configuration level. Since the original experiments involve large public trip records, city-specific geospatial files, processed tensors, trained checkpoints, and server-specific paths, the repository is organized as modular code components rather than as a one-click executable package.

## Repository Structure

```text
code/
├── CC-STMT/
│   ├── common/
│   ├── NYC/
│   ├── Chicago-A/
│   └── Chicago-B/
└── DP-OTM/
    ├── common/
    ├── NYC/
    ├── Chicago-A/
    ├── Chicago-B/
    └── explain.md
```

The `CC-STMT` directory contains the prediction-related implementation. It includes the shared model definition and the city-specific data processing, training, and evaluation scripts.

The `DP-OTM` directory contains the dispatch-related implementation. It includes the shared dispatching logic, mixed-request environment construction, vehicle-request state representation, and city-specific dispatching scripts.

## Scope of the Released Code

The released code focuses on the proposed method and its internal component analysis. It includes:

```text
CC-STMT model definition
City-specific prediction data processing scripts
City-specific CC-STMT training scripts
City-specific CC-STMT evaluation scripts
DP-OTM dispatching environment construction
Mixed on-demand and advance request construction
Vehicle and order state management
Destination-aware matching cost construction
Unbalanced optimal transport based matching prior
Linear assignment based executable dispatching
Component ablation settings for destination opportunity and destination risk
```

The released code does not include external forecasting baseline implementations or external dispatching baseline implementations. The paper reports comparisons with baseline methods, but the released repository is limited to the proposed framework and its internal component structure.

## Data and File Dependencies

The original experiments use public trip records and public spatial or contextual data. Large raw datasets are not stored directly in this repository. Users should obtain the raw data from the public sources described in the manuscript and prepare the corresponding city-specific input files.

The prediction module expects city-specific raw files and produces processed tensors such as:

```text
demand_tensor_full.npy
speed_tensor_full.npy
enroute_tensor_full.npy
weather_tensor.npy
adj_spatial.npy
adj_semantic.npy
dist_matrix.npy
zone_mapping.csv
```

The dispatching module expects the processed trip records, distance matrix, zone mapping, and prediction outputs generated by the CC-STMT evaluation stage. Typical prediction outputs used by the dispatching module include:

```text
test_predictions_expected.npy
test_predictions_mu.npy
test_predictions_pi.npy
test_predictions_theta.npy
test_predictions_v_hat.npy
test_ground_truth.npy
```

These files connect the prediction module and the dispatching module. The predicted demand intensity and low-opportunity risk fields are used to construct destination opportunity and destination risk terms in DP-OTM.

## Relationship Between CC-STMT and DP-OTM

The framework follows a prediction-dispatching pipeline.

First, CC-STMT processes multi-source urban inputs, including historical demand, traffic-speed states, en-route supply feedback, weather or contextual features, and dual graph structures. It outputs probabilistic prediction fields, including demand intensity, low-opportunity risk, demand dispersion, and an auxiliary speed prediction.

Second, DP-OTM uses the prediction outputs to evaluate candidate request destinations. For each vehicle-request pair, the dispatching cost considers current service factors and predicted destination-side factors. The final assignment is generated through an unbalanced optimal transport prior and a one-to-one linear assignment step.

This design allows the forecasting model to support dispatching decisions directly, rather than serving only as a standalone demand-count prediction model.

## City Settings

The repository separates the three experimental settings into city-specific folders:

```text
NYC
Chicago-A
Chicago-B
```

The three settings differ in raw data sources, spatial zoning systems, temporal resolution, preprocessing assumptions, and experimental configuration. Therefore, each city folder contains its own scripts for data processing and experiment preparation. Shared model and dispatching logic are placed in the corresponding `common` directories.

## Reproducibility Notes

The code is released as a structured implementation reference for the proposed method. To reproduce the full experimental workflow, users need to prepare the public raw data, city boundary files, road-network or distance information, weather data, processed tensors, and trained model checkpoints according to the data descriptions in the manuscript.

Some file paths in the original experimental scripts may need to be adapted to the local machine or server environment. The original experiments were performed under a fixed project directory layout, while this repository separates the implementation into clearer method-level folders.

Exact numerical reproduction requires consistency in:

```text
raw data version
spatial zoning files
preprocessing rules
temporal split
random seed
fleet size
request sampling rule
advance-request construction rule
prediction checkpoint
dispatching cost weights
software environment
```

## Excluded Content

The repository intentionally excludes the following content:

```text
external forecasting baseline code
external dispatching baseline code
large raw public datasets
large intermediate tensors unless explicitly provided
large trained baseline weights
temporary cache files
notebook checkpoints
server-specific backup files
```

This keeps the repository focused on the proposed method and avoids distributing unnecessary large files or unrelated experimental artifacts.
